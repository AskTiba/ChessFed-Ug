## 2. Authentication & Security (NextAuth)
The application has built-in page middleware protection and secure server actions. To ensure NextAuth works securely in production:

*   [ ] **Generate a secure `NEXTAUTH_SECRET`:**
    In local development, this is a placeholder. For Vercel, generate a cryptographically strong 32-character random string (e.g., run `openssl rand -base64 32` in your terminal) and copy it.
*   [ ] **Set the `NEXTAUTH_URL`:**
    Define the canonical URL of your deployed application (e.g., `https://chesshub-ug.vercel.app`) in Vercel so NextAuth can manage secure redirects and callbacks.

## 3. Serverless Execution Constraints (10-Second Vercel Limit)
> [!WARNING]
> **Vercel's Serverless Functions on the free tier have a strict 10-second limit.**
> The FIDE sync API takes 1–2 minutes because it runs ~8,600 individual `upsert` queries over the network to Supabase. This will cause a timeout on Vercel.

*   [ ] **Optimize the Sync Batch Insertion (Recommended Next Step):**
    We should refactor `/app/api/fide/sync/route.ts` to use Prisma's `createMany({ skipDuplicates: true })` inside larger batches of 1,000 players. This cuts the database roundtrips from 8,600 down to **less than 10**, dropping the execution time from 2 minutes to **under 3 seconds**!
*   [ ] **Manual Trigger:**
    Alternatively, trigger the sync locally via a CLI script or GitHub Action that isn't bound by Vercel's serverless timeouts.

## 5. Pre-Deployment Build Validation
Before pushing to Git to trigger a Vercel deployment, verify that everything compiles with no TypeScript or Turbopack errors:

*   [ ] **Run a test production build locally:**
    ```bash
    npm run build
    ```
    This compiles the Prisma Client and runs the Next.js production build compiler. Resolving TypeScript schema mismatches or import errors locally is much faster than waiting for Vercel build failures.
